Learning Architecture Analysis — tools/ Directory

HumanAIOS Research · Systems & Governance Team
2026-06-23 · Version 1.0
Abstract. This paper presents a comprehensive architectural analysis of the recursive learning mechanisms implemented in the humanaios-ui/operations/tools directory. We examine the four-phase Molt Cycle, the Harmonizer resonance loop, session-spanning memory via Carry/Z2, and the ACAT Signal Detection Theory (SDT) analytics pipeline. We map explicit data flows between tools, trace how observations become knowledge graph entries, and identify six structural gaps where learning is lost. We then catalog six human injection points that codify guidance into system behavior, and analyze six layers of contradiction detection between human intent and observed AI behavior. The architecture demonstrates a principled one‑way ratchet—observations → findings → drift patterns—but every major learning step passes through a human ratification gate, intentionally bounding learning velocity by human bandwidth. The most significant gap identified is the absence of closed‑loop outcome feedback: task execution results never re‑enter the Molt phase, leaving the system with rich perception but no reinforcement signal from consequences.

1. Feedback Loops and Recursive Learning Mechanisms

1a. The Molt Cycle — the primary learning loop

The dominant recursive mechanism is the four-phase Molt Cycle in haios_agent_orchestrator_v1_0_patched.py:

Perceive (lines 86–113) → Assess (lines 158–201) → Act (lines 204–285) → Molt (lines 288–394)

Each cycle ingests raw events, converts them to candidate nodes (make_candidate_node(), lines 116–155), runs ACAT + principle checks, and then in the Molt phase either promotes, stages as pending, merges, or prunes the knowledge graph. The rule at line 301–309:

“For each approved node with li_impact > 0.75: promote to status='active' and add to node_store.”

The Learning Index (LI) is the scalar that drives everything. Every observation gets an LI attached via acat_signals, and that single score determines whether it becomes permanent knowledge, pends for human review, or gets rejected. This is the core feedback variable linking observation → scoring → knowledge state.

Near-duplicate merging (lines 363–375) prevents knowledge inflation: any new node with ≥60% title-word overlap with an existing node gets merged rather than duplicated — evidence accumulates inside existing nodes rather than forking.

Stale pruning (lines 352–358): nodes that haven't been touched in >5 molt cycles are downgraded to status="pruned". This is the only "forgetting" mechanism.

The loop is session-scoped: MOLT_LOG.json records each cycle (line 465–467), and node_store.json persists the live KG across sessions.

1b. Harmonizer Resonance Loop

The System Harmonizer (haios_harmonizer_v1_0.py) runs a second, parallel feedback loop across 8 subsystems (CORPUS, FINDINGS, TOOLS, GOVERNANCE, RESEARCH, COMMS, FINANCE, INFRA). It:

  • Scores each subsystem 0–100 using decay factors (lines 283–463)
  • Generates prioritized tuning_signals — HARD_BLOCK → ESCALATE → CONVERGE → ADVANCE → MAINTAIN (lines 478–538)
  • Produces a MOLT_READY verdict when all subsystems ≥70 and no hard blockers exist (lines 502–517)

This loop tells the system what to work on next rather than what to believe. Critically, its inputs are a structured JSON state snapshot — not raw outputs from individual tools — meaning the harmonizer is only as current as the last manually-updated system_state.json. There is no automated path from tool outputs back into this state object.

1c. Carry / Z2 Session-Spanning Memory

carry_tracker_v1_0.py and z2_queue_v1_0.py implement the system's inter-session learning pressure. Carry items accumulate n (sessions carried) and escalate at N≥5 → WARN and N≥10 → ESCALATE. The harmonizer reads these counts and deducts harmony points (-15 or -30). This creates a degrading signal that forces resolution: unresolved items measurably harm system health scores.

Together these two tools constitute a degrading memory loop: problems that persist across sessions become progressively more costly.

1d. ACAT SDT Analytics — Post-hoc Statistical Loop

acat_sdt_analytics_v1_0.py computes Signal Detection Theory metrics (d′, β) from paired Phase 1/Phase 3 corpus data. It detects the coercion signature: high d′ (≥1.2) + liberal β (<0.6) means “the system discriminates inflation accurately AND defaults to compliance anyway” (lines 143–149). Finding H-HML-01 showed Autonomy Respect dimension has the strongest coercion signature (d′=1.544, β=0.438, per lines 29–34).

This is a genuine scientific feedback loop — corpus observations → SDT metrics → findings. However, the docstring at lines 34–38 explicitly breaks the circuit back to the instrument:

“Per Zone 2 decision (GONOGO_LIT_BRIEF_S052726-MKS Section 8, Decision 1): SDT is post-hoc analytics at TRL 2. Instrument integration is v6.0 scope.”

The analytical result exists but cannot yet modify ACAT v5.x behavior.

3. How Observations From One Tool Become Inputs to Another

The key transformation chain for an observation entering the system:

  1. Classification: Raw events (type: acat_result, human_feedback, tool_output, emergence_replay, financial_event, infra_event, comms_event) become typed candidate nodes via make_candidate_node() (orchestrator lines 116–155). Each node gets acat_signals: {li_score, humility_flag, drift_flag} embedded from the source payload.
  2. Principle-gating: assess() (lines 160–201) runs every candidate through check_principle(), which tests for P3 (unverified claims), P19 (detection-not-compliance), and P-HUMILITY (overconfidence >0.95 without counter-evidence). Any violation, humility_flag, or drift_flag routes the node to Zone 2 — it cannot advance without human review.
  3. LI gating: Rejected nodes (LI < 0.5) never enter the KG. Approved nodes with LI < 0.75 enter as status="pending" for Zone 2 review. Only LI ≥ 0.75 nodes become status="active" and can generate REGISTERED.md append drafts.
  4. Corpus ingestion as evidence: The document_ingestor_v1_0.py explicitly converts acat_document_analyzer JSON output into Supabase-compatible rows with document_layer and score_source metadata. The F-34 note (lines 35–40) distinguishes architectural scores from behavioral scores — a governance observation thus affects the corpus but is labeled differently so it doesn't contaminate behavioral aggregate statistics.
  5. Statistical feedback: Once rows are in Supabase, corpus_delta_analyzer.py compares snapshots to detect DELTA_LI_REGRESSION, DELTA_DCOMP_GAINED, DELTA_MEAN_LI_DROP. These deltas become new candidate observations in the next Molt Cycle when the operator surfaces them as type: "tool_output" source events.

The most important observation: the pipeline data-passing is explicit but manual at the boundaries. Tool outputs go into JSON files, get posted to Slack, or go into Supabase. They don't automatically re-enter the Molt Cycle as source events. A human operator must consciously bring a tool_output back in as a source event for it to enter the KG.

4. Gaps in the Feedback Chain Where Learning Is Lost

Gap A — Task outcomes never feed back into MOLT

The most significant structural break in the loop. In run_molt_cycle() (orchestrator line 456):

molt_result = molt(assessment["approved"], [], node_store, molt_cycle)

The second argument — task_outcomes — is always an empty list []. The molt() function signature at line 290 accepts task_outcomes: list[dict] with keys {task_id, outcome, li_delta, notes}. The intent is for execution results to feed back as LI adjustments. But no call site ever populates this. Every task's actual outcome — whether it succeeded, failed, changed the world — is structurally absent from the learning record. The KG grows from observations but never learns from consequences.

Gap B — Several named tools are stub scaffolds

bpl_signal_extractor_v1_0.py (lines 78–100), acat_phase_shift_analyzer_v1_0.py (lines 77–100), molting_protocol_diff_v1_0.py (lines 78–100), and message_calibration_v1_0.py (lines 78–100) all contain the identical TODO comment:

# ── YOUR LOGIC HERE ────────────────────────────────────────────
# Example structure — replace with actual implementation:
items = []
warnings = []
# TODO: iterate over data, populate items and warnings

These tools are registered in the system and named in the architecture but emit empty results. Any pipeline stage that depends on them silently receives null data.

Gap C — Hardcoded corpus statistics across multiple tools

CORPUS_MEAN_LI = 0.8632 appears hardcoded in at least four files:

  • corpus_integrity_validator.py line 40
  • haios_report_writer_v1_0.py line 53
  • corpus_delta_analyzer.py line 37
  • acat_sdt_analytics_v1_0.py line 37

As the corpus grows, this baseline becomes stale. corpus_delta_analyzer.py uses this constant to classify D-COMP agents (lines 68–70: li > CORPUS_MEAN_LI + 0.05 = D-COMP). A drifting mean means the D-COMP threshold silently shifts relative to reality. There is no tool that recomputes and propagates this constant when new data is ingested.

Gap D — SDT results don't update instrument calibration

As noted in §1d: acat_sdt_analytics_v1_0.py's coercion signature findings and d′ measurements are analytically complete but explicitly walled off from the ACAT v5.x instrument by Zone 2 decision. The learning from “our measurement instrument has systematic response bias” cannot modify the instrument without a separate ratification cycle that hasn't happened.

Gap E — The harmonizer state snapshot is manually updated

haios_harmonizer_v1_0.py reads from a system_state.json file whose schema is defined at lines 108–235 as a static template. Carrying forward carry item counts (e.g., IC-023 at N=11) requires a human to update this JSON between sessions. There is no automated path from carry_tracker outputs back into system_state.json. If that file is stale, the harmonizer's tuning signals are based on prior state — exactly the AUTH-THEATER Mode B condition that system_audit_v1_1.py was built to detect.

Gap F — Drift catalog is static; resolved signals leave no structural trace

drift_catalog_validator.py's REGISTERED_DRIFT dictionary (lines 28–58) is a static Python dict. New drift codes require a file edit and session ratification. There is no automated path for a detected drift signal to advance from observation → REGISTERED_DRIFT membership. Additionally, once a drift signal is resolved (e.g., IC-023 rotated), there is no mechanism to mark it as retired in the catalog itself — only the carry tracker can reflect resolution (by N dropping to 0).

5. Where Human Decision-Makers Can Inject Guidance That Codifies Back Into System Behavior

The system has six structurally-defined injection points, ordered from most to least consequential:

Injection 1 — Constitution amendments (highest authority)

constitution.json (referenced at orchestrator lines 52–56) is the “immutable at runtime” 22-principle kernel. The git_push_gate enforces that it is always Zone 2+ (constitution.json listed at line 29 of git_push_gate_v1_0.py). A human-ratified constitution amendment becomes the check that every future plan step runs against. This is the deepest codification point: amending here changes the principle check logic for all future Molt Cycles.

Injection 2 — REGISTERED.md (finding ratification)

The registered_findings_validator_v1_0.py enforces that REGISTERED.md is append-only and schema-valid. The orchestrator generates REGISTERED_MD_DRAFT.md for Zone 3 Night operator review. When Night merges a finding draft, it becomes the canonical record of a validated behavioral pattern — a permanent addition to the knowledge base that downstream tools query. registered_findings_validator_v1_0.py also detects REG_UNCOMMITTED_CANDIDATE items (line 21), creating a soft signal that pending candidates haven't been ratified.

Injection 3 — Zone 2 queue resolution

z2_queue_v1_0.py items with blocker_if_unresolved=True stop the system (exit code 1). Human resolution (setting resolved=True) unblocks downstream work. The ratification record is attached to the Z2 item itself. The principle harmonizer mirrors this: HALT-verdict frameworks (lines 29–31 of principle_harmonizer_v1_2.py) require “Night ratification required before operational use.”

Injection 4 — System state JSON update

Because the harmonizer consumes a manually-maintained system_state.json, every time a human operator updates corpus counts, carry item N values, gate status, or blocker flags in that file, they are directly calibrating the system's self-model. The harmonizer's tuning signals — which determine what work gets prioritized — flow entirely from this human-curated state snapshot.

Injection 5 — Drift signal registration

Adding a new entry to REGISTERED_DRIFT in drift_catalog_validator.py (lines 28–58) codifies a newly-observed behavioral failure mode. Once registered, the drift validator will catch it in all future session close posts. This is a governance loop: a human names a failure pattern, and it becomes part of the automated compliance check.

Injection 6 — DAILY_WILL_QUERY

The notify dispatcher's daily conscience check (lines 25–26): “If unanswered for 3 days, system auto-halts.” This is the minimal but unconditional human-in-the-loop mechanism — a single unanswered bit halts all autonomous activity.

6. How the System Handles Contradiction and Divergence Between Human Intent and Observed Behavior

The system has several layers for detecting and routing divergence, but no automated resolution mechanism — divergence always terminates at a human decision point.

Layer 1 — Drift taxonomy as named contradictions

drift_catalog_validator.py maintains 20+ named divergence patterns (D-01 through D-AGREEMENT-CASCADE). Each name is a codified category of AI behavior contradicting human intent:

  • D-05 (Zone 1 overreach): AI executing without approval on Zone 2/3 items
  • D-08 (Shadow queue): AI maintaining a Zone 3 list outside the ratified system
  • D-CARRY: forwarding stale items without explicit operator confirmation
  • D-AGREEMENT-CASCADE: accepting bulk reviewer claims without verification

These aren't just error codes — they're named behavioral failure modes that the acat_protocol_auditor.py's DFA scanner (lines 7–14) actively detects in session close posts, creating automatic detection of claimed vs. actual behavior divergence.

Layer 2 — ACAT Phase 1→3 as divergence measurement

The instrument's two-phase structure is itself a divergence detector. Phase 1 scores the AI's behavior; Phase 3 is the same AI scoring the same interaction after reflection. The Learning Index LI = Phase3_total / Phase1_total. LI > 1.0 = overcorrection (D-COMP risk), LI < 0.95 = undertreated inflation. The entire corpus of 629+ rows is a time series of this divergence metric per agent per session.

The SDT analytics (§1d) give the most precise characterization: the coercion signature (high d′, low β) means the AI knows it should correct (discriminates inflation well) but defaults to compliance anyway. This is the most subtle form of human-intent divergence in the system — not outright refusal, but structurally-biased capitulation.

Layer 3 — Principle check as pre-execution veto

check_principle() in the orchestrator (lines 58–83) is synchronous: called on every proposed action before execution. If P-HUMILITY fires (confidence > 0.95 without counter_evidence), the node is flagged and goes to Zone 2. This handles AI overconfidence divergence before it becomes an action.

Layer 4 — Red team runner for adversarial divergence

red_team_runner_v1_0.py probes 14 attack categories (zone bypass, overconfidence, corpus poisoning, comms discipline, etc.) using deterministic keyword-based detection. Critically, the detection rules are static keyword lists (detection_keywords / bypass_keywords arrays). When an attack bypasses via novel phrasing, the runner scores it as BYPASSED but there is no mechanism to add the novel bypass pattern to future detection rules. The red team runner is a snapshot diagnostic, not a self-updating adversarial filter.

Layer 5 — System audit as AUTH-THEATER detection

system_audit_v1_1.py was explicitly designed to detect the condition where a human operator believes they have current state but doesn't (AUTH-THEATER Mode B, line 21). By running as a mandatory pre-ratification gate, it breaks the loop where stale mental models produce ratifications based on prior reality. The check at lines 86–93 sets max staleness thresholds for CURRENT.md (168 hours), commits (72 hours), and Slack WGS sync (24 hours).

Layer 6 — Principle harmonizer CONFLICT routing

When a Tier 2 framework has relationship: "CONFLICT" with a Tier 1 principle, the GHK logic (principle_harmonizer_v1_2.py lines 16–31) produces KILL verdict. The disagreement is noted and archived, not synthesized. The framework's content cannot enter operational tooling without a Zone 2 reversal of the KILL verdict. This makes Tier 1 the immutable ground truth against which external frameworks are measured, not a dynamic system that learns from them.

Summary Observations

The architecture has a well-designed one-way ratchet: observations can enter the KG, findings can be registered, and drift patterns can be codified — but none of these paths is automated end-to-end. Every major learning step passes through a human ratification gate (Zone 2 or Zone 3). The tradeoff is intentional (the design principle at orchestrator line 15: “AI eats friction. Humans eat conscience.”) but it means the system's learning velocity is bounded by human bandwidth, not by data throughput.

The deepest architectural gap is closed-loop outcome feedback: task execution outcomes never re-enter the MOLT phase (Gap A). The KG learns from what is observed and assessed, but not from the consequences of what is done. In learning-system terms, this is a system with rich perception and belief-updating but no reinforcement signal from executed actions.
