# SonarCloud Baseline Auto-Capture — Automation Guide

**Workflow:** `.github/workflows/sonarcloud-baseline-auto.yml`  
**Trigger:** Automatic on every push to `main` branch  
**Duration:** ~5 minutes per run  
**Manual steps:** None (completely automated)

---

## Overview

The SonarCloud Baseline Auto-Capture workflow is a fully automated GitHub Actions pipeline that:

1. ✅ Runs SonarCloud scan automatically
2. ✅ Fetches metrics from SonarCloud API
3. ✅ Auto-creates GitHub issues for BLOCKER violations
4. ✅ Updates METRICS_BASELINE.md with fresh metrics
5. ✅ Commits changes back to main branch

**Result:** Your baseline metrics are captured and tracked automatically after every commit. No manual intervention required.

---

## How It Works

### Step 1: Trigger (Automatic)

Every time you push to `main`, the workflow automatically kicks off:

```bash
git commit -m "some change"
git push origin main
# Workflow automatically triggers → scan begins
```

Or use workflow_dispatch to run manually:

```
GitHub Actions → sonarcloud-baseline-auto → "Run workflow" button
```

### Step 2: SonarCloud Scan

The workflow runs the SonarQube scan action:

```yaml
- name: Run SonarQube Scan
  uses: sonarsource/sonarqube-scan-action@v3
  env:
    SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
    SONAR_HOST_URL: ${{ secrets.SONAR_HOST_URL }}
```

**Requires:**
- ✅ `SONAR_TOKEN` secret (GitHub → Settings → Secrets)
- ✅ `SONAR_HOST_URL` secret (should be `https://sonarcloud.io`)

**If secrets missing:** Workflow silently skips (no errors, just a notice)

### Step 3: Fetch Metrics from SonarCloud API

Python script queries SonarCloud API for:

```python
# Quality Metrics
- reliability_rating
- security_rating
- sqale_rating (maintainability)
- coverage
- duplicated_lines_density
- bugs
- vulnerabilities
- code_smells
- ncloc (lines of code)
- sqale_index (technical debt)

# Issues
- Issues filtered by severity (BLOCKER, CRITICAL, MAJOR, MINOR, INFO)
- Component (file path)
- Line number
- Rule name
- Description
```

**API Calls:**
```
GET /api/issues/search
GET /api/measures/component
```

### Step 4: Auto-Create GitHub Issues

For each BLOCKER violation found:

1. Checks if issue already exists (by title) to avoid duplicates
2. Creates new GitHub issue with:
   - Title: `[Code Quality] BLOCKER: {rule name}`
   - Labels: `blocker`, `code-quality`
   - Link back to SonarCloud
   - Remediation checklist
3. Limited to 10 issues per run (to avoid overwhelming GitHub)

**Example issue created:**
```
Title: [Code Quality] BLOCKER: SQL Injection vulnerability
Labels: blocker, code-quality
Body: 
  Rule: SQL Injection vulnerability
  Component: acat/database.py
  Line: 42
  [Remediation checklist...]
  [Link to SonarCloud]
```

### Step 5: Update METRICS_BASELINE.md

Automatically updates `docs/METRICS_BASELINE.md` with:

- **Timestamp:** Current date (e.g., "2026-07-22")
- **Quality Grades:** A/B/C/D/E for each dimension
- **Issue Counts:** Updated severity breakdown
- **Coverage:** XX%
- **Duplication:** XX%

**No manual editing needed** — all `[PENDING]` placeholders automatically filled in.

### Step 6: Commit & Push

Workflow commits changes back to main:

```bash
git config user.name "sonarcloud-baseline-bot"
git config user.email "sonarcloud-baseline@github.com"
git add docs/METRICS_BASELINE.md
git commit -m "chore(metrics): auto-update baseline from SonarCloud scan"
git push origin main
```

**Commit message includes:**
- Timestamp of capture
- What was updated
- Auto-generated by bot attribution

## Troubleshooting

### Workflow doesn't run

**Check:** Secrets are configured
```
GitHub Settings → Secrets and variables → Actions
Must have: SONAR_TOKEN, SONAR_HOST_URL
```

**Check:** Workflow YAML is valid
```
Actions tab → Check for syntax errors
```

### Issues not being created

**Most likely:** They already exist (duplicate check working)
```
Search GitHub issues for "BLOCKER" label
View issue dates — if recent, duplicates prevented
```

**Check logs:** Actions → Run logs → "Create GitHub issues" step
```
Look for: "Issue already exists" messages
```

### Baseline not updating

**Check logs:** Actions → Run logs → "Update METRICS_BASELINE.md" step
```
Verify: "✓ Updated METRICS_BASELINE.md with latest metrics"
```

**Common issues:**
- SonarCloud API call failed (check secrets, API rate limit)
- File path wrong (should be `docs/METRICS_BASELINE.md`)
- Permissions missing (workflow needs `write: contents`)

### Commits piling up

If bot commits too frequently, you can throttle by:

```yaml
# In workflow, change trigger to:
on:
  push:
    branches: [main]
    paths:  # Only trigger on code changes, not on bot commits
      - '**.py'
      - '**.js'
      - '**.yml'
      # Add all code paths except docs/METRICS_BASELINE.md
```

## Best Practices

### ✅ Do This

- Let the automation run — it's non-blocking
- Review metrics weekly (in METRICS_BASELINE.md)
- Assign created BLOCKER issues to developers
- Monitor GitHub Actions logs for errors

### ❌ Don't Do This

- Manually edit METRICS_BASELINE.md in the "Pending" sections (automation overwrites them)
- Disable the workflow (it's non-intrusive)
- Ignore BLOCKER issues (they're the highest priority)
- Push secrets as plain text (use GitHub Secrets)

## Security Notes

### What the workflow can access:

```
✓ SONAR_TOKEN (read-only from secrets)
✓ GITHUB_TOKEN (for creating issues, writing commits)
✓ Code repository (read-only checkout)
```

### What it CANNOT access:

```
✗ Other secrets
✗ Production systems
✗ External services (except SonarCloud API)
```

### Bot commits:

```
User: sonarcloud-baseline-bot
Email: sonarcloud-baseline@github.com
Signature: Auto-generated bot commits, not user commits
```

## Customization

### Adjust issue creation limit

Edit `.github/workflows/sonarcloud-baseline-auto.yml`:

```yaml
# Change this line (currently: 10)
for issue in blocker_issues[:10]:  # Create max 10 issues per run
```

### Add more metrics to capture

Edit Python script in workflow:

```python
measure_keys = [
    # Add more SonarCloud metrics here
    "complexity",
    "cognitive_complexity",
    "test_success_density"
]
```

### Change commit message format

Edit workflow step:

```yaml
git commit -m "custom message format"
```

### Trigger on schedule instead of push

```yaml
on:
  schedule:
    - cron: '0 9 * * *'  # Every day at 9 AM UTC
```
